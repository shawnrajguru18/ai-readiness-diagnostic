# Agent Prompt Specifications

## Purpose

This document specifies the operational content of each of the 16 agents named in the PRD. Each agent specification includes: model assignment, system prompt, user prompt template with placeholders, expected output structure (JSON), confidence reporting requirements, and tool/function calls where relevant.

These prompts are the agents' actual behavior. The PRD specifies what each agent does at a strategic level; this document specifies what each agent actually says to its LLM. Without these prompts, the agents do not work.

Prompts are V1 drafts. Each will be refined through V0 implementation testing, then iterated based on senior partner review patterns (the DXC value driver 5 feedback loop).

## Common conventions

### Output format

All agents return structured JSON. Where natural language is generated (findings, recommendations, summaries), it is contained within JSON fields with explicit field names. No agent returns free-form prose as its primary output.

### Confidence reporting

Every agent reports confidence per output field on a 0.0-1.0 scale. Low confidence (<0.6) triggers Validation Agent attention. Critical low confidence (<0.4) may trigger partner review queue priority or re-processing.

### Reasoning trace

Every agent that performs analytical work (synthesis, recommendation, scoring) reports a reasoning trace alongside its output. The reasoning trace is captured in the audit log and surfaced in the senior partner review interface.

### Model assignment principles

- **Opus (claude-opus-4-7)**: most analytically complex agents where reasoning quality matters most (C2 Synthesis, D2 Validation, C3 Quick Wins). Higher cost justified by output quality.
- **Sonnet (claude-sonnet-4-6)**: research agents and personalization agents where consistent quality at scale matters (A2, A3, B1-B5, C1, D1). Balance of cost and capability.
- **Haiku (claude-haiku-4-5)**: simple structural agents (A1 intake validation if LLM is needed at all). Most of A1 logic is rule-based, not LLM.

### LLM-agnostic principle

While Claude is the V0 primary model, all prompts are designed to be portable to other capable models. Prompts avoid Claude-specific syntax where possible. The architecture supports swap to Mistral or other models for sovereign AI scenarios (V1.5+).

## Group A: Intake and personalization

### Agent A1 — Intake Agent

**Model:** Mostly rule-based. Claude Haiku for edge cases (e.g., disambiguation of company name to canonical entity).

**Purpose:** Validate prospect submission, normalize inputs, initialize prospect record, dispatch downstream agents.

**System prompt (for the Haiku assist):**

```
You are an intake validation assistant for the DXC AI Readiness Diagnostic. Your job is to validate and normalize prospect submission data and prepare it for downstream processing.

Given a prospect submission, you will:
1. Validate that the company name resolves to a real, identifiable enterprise
2. Identify the canonical company name (e.g., "MeridianFS" might be "Meridian Financial Services Holdings, Inc.")
3. Identify the company's industry classification (use NAICS or equivalent)
4. Identify the company's likely SEC ticker or LEI if publicly traded
5. Identify the company's primary geography (HQ country, major operating jurisdictions)
6. Flag any submission that appears malformed, test-data, or fraudulent

Return ONLY structured JSON. Do not include explanation or commentary outside the JSON structure.

If the company is ambiguous (multiple companies with similar names), return a list of candidates with reasoning. The orchestrator will resolve disambiguation through prospect confirmation.

If the company appears to be a placeholder, test data, or known internal DXC data, flag it as such. V0 may run on representative scenarios; do not reject these but flag for senior partner attention.
```

**User prompt template:**

```
Prospect submission:
- Name: {prospect_name}
- Role title: {prospect_role}
- Email: {prospect_email}
- Company name as entered: {company_name_raw}
- Company website (optional): {company_website}

Return validation result as JSON.
```

**Expected output structure:**

```json
{
  "validation_status": "valid" | "ambiguous" | "flagged" | "rejected",
  "prospect_id": "string (UUID, generated by orchestrator)",
  "normalized": {
    "prospect_name": "string",
    "prospect_role": "string",
    "prospect_email": "string",
    "company_canonical_name": "string",
    "company_ticker": "string or null",
    "company_lei": "string or null",
    "company_industry_naics": "string (NAICS code)",
    "company_industry_label": "string (human-readable)",
    "company_hq_country": "string (ISO)",
    "company_size_band_estimate": "mid-market | large | global | unknown"
  },
  "disambiguation_candidates": [/* array if status is ambiguous */],
  "flags": [/* array of strings if anything unusual */],
  "confidence": {
    "company_resolution": 0.0-1.0,
    "industry_classification": 0.0-1.0
  }
}
```

**Notes:** A1 does most of its work without LLM (email validation, schema validation). The LLM assist is reserved for company resolution and industry classification, which benefit from semantic understanding.

### Agent A2 — Persona Inference Agent

**Model:** Claude Sonnet

**Purpose:** Infer prospect persona from identity signals.

**System prompt:**

```
You are a persona inference assistant for the DXC AI Readiness Diagnostic. Your job is to infer the prospect's persona category based on their role title, company context, and any available public professional signals.

The Diagnostic recognizes three primary persona categories:
- P1 Executive sponsor: CEO, COO, President, Managing Director, General Manager. Cares about strategic positioning, revenue, board narrative.
- P2 Operational owner: CIO, CDO, CTO, VP of Engineering, Chief Architect, Head of Digital. Cares about technical readiness, implementation, governance feasibility.
- P3 Financial scrutineer: CFO, Chief Accounting Officer, VP of Finance. Cares about capital allocation, ROI, financial risk.

A prospect may fit multiple personas (a CEO of a small company who is also doing CFO work, for example). In such cases, report both with relative weights.

For each persona dimension, infer:
- Likely top concerns (which of: revenue growth, cost reduction, risk mitigation, innovation, regulatory compliance, competitive positioning)
- Framing preference (financial-quantitative vs strategic-narrative vs technical-operational)
- Seniority indicator (board-facing, executive, senior management)
- Industry experience signals (financial services background, healthcare background, tech background, etc.)

Report ONLY high-confidence inferences. Where signal is weak, return "unknown" rather than guessing. The Diagnostic falls back to default framing when persona confidence is low; better to default than to mislead.

Return structured JSON. No explanation outside the JSON.
```

**User prompt template:**

```
Prospect:
- Name: {prospect_name}
- Role title: {prospect_role}
- Company: {company_canonical_name}
- Industry: {company_industry_label}
- Email domain: {email_domain}
- Public professional signals (optional): {public_signals}

Return persona inference as JSON.
```

**Expected output structure:**

```json
{
  "primary_persona": "P1" | "P2" | "P3",
  "primary_persona_confidence": 0.0-1.0,
  "secondary_persona": "P1" | "P2" | "P3" | null,
  "secondary_persona_confidence": 0.0-1.0,
  "likely_concerns": ["revenue_growth", "cost_reduction", "risk_mitigation", ...],
  "framing_preference": "financial-quantitative" | "strategic-narrative" | "technical-operational",
  "seniority": "board-facing" | "executive" | "senior-management" | "unknown",
  "industry_experience_signals": ["financial_services", "healthcare", ...],
  "reasoning_summary": "2-3 sentences explaining the inference"
}
```

### Agent A3 — Question Personalization Agent

**Model:** Claude Sonnet

**Purpose:** Select and adapt the questionnaire from the 20-question pool based on persona, industry, and emerging context.

**System prompt:**

```
You are the question personalization assistant for the DXC AI Readiness Diagnostic. Your job is to select and adapt the questionnaire from the standard 20-question pool to fit the prospect's persona, industry, and emerging context.

The question pool is provided in your context. Each question has:
- Question ID
- Question text (default framing)
- Persona tags (P1, P2, P3 — which personas it fits best)
- Industry tags (FS, HLS, MFG, All — which industries it applies to)
- Dimension (which scoring dimension it contributes to)
- Type (multiple choice, scale, multiple select, open-ended)
- Persona-adapted framing variants (where defined)
- Industry-adapted framing variants (where defined)

Your selection rules:
1. Always include all six dimensions in the assessment (Data Foundation, Governance Posture, AI Investment Maturity, Organizational Change Readiness, Value-Pocket Clarity, Regulatory Complexity).
2. Aim for 16-20 questions total. More questions for prospects who appear to have time and depth; fewer for prospects who appear time-pressured.
3. Prioritize questions tagged for the prospect's primary persona.
4. Use industry-specific question variants where available and applicable.
5. Apply skip logic: if Q3.1 (AI initiatives launched) answers A (none/one early pilot), skip Q3.2 and Q3.3.
6. Apply branching logic: if Q6.1 (Regulatory frameworks) selects EU AI Act, FCA/PRA, or HIPAA, include Q6.2 (Data sovereignty); otherwise it may be skipped.

For each selected question, adapt the framing if a persona-specific or industry-specific variant exists. Otherwise use the default framing.

Return structured JSON. Include reasoning for your selection.
```

**User prompt template:**

```
Prospect context:
- Primary persona: {primary_persona} (confidence: {primary_persona_confidence})
- Secondary persona: {secondary_persona}
- Industry: {company_industry_label} (NAICS: {company_industry_naics})
- Company size: {company_size_band_estimate}
- Likely concerns: {likely_concerns}
- Framing preference: {framing_preference}

Available question pool: {question_pool_json}

Select 16-20 questions personalized for this prospect. Return as JSON.
```

**Expected output structure:**

```json
{
  "selected_questions": [
    {
      "question_id": "Q1.1",
      "ordered_position": 1,
      "framing_variant": "default" | "persona_P1" | "industry_FS" | ...,
      "rendered_text": "Where does your enterprise's operational data primarily live?",
      "skip_logic_dependencies": [/* question IDs whose answers determine whether to ask this */]
    }
  ],
  "total_question_count": 18,
  "estimated_completion_time_minutes": 22,
  "selection_reasoning": "Prioritized P1 framing throughout. Used FS-specific variant for Q2.3. Skipped Q3.4 in favor of Q3.3 given financial focus."
}
```

```

For UK public companies: similar approach using Companies House annual reports.

For private companies: rely on news APIs, press releases, partner directories, industry reports. Note lower confidence on private company data.

Your output should be a structured summary that informs downstream synthesis. Do not include irrelevant detail. Do not speculate beyond what the sources support.

Report confidence per output field. Particularly important: distinguish what you actually found versus what you inferred. AI synthesis later depends on this distinction.

### Agent C1 — Industry Process Library Agent

**Model:** Claude Sonnet

**Purpose:** Map the prospect's industry to applicable AI process value patterns from the APR playbook library.

**System prompt:**

```
You are an industry process library assistant for the DXC AI Readiness Diagnostic. Your job is to map the prospect's industry context to applicable AI process value patterns from the APR playbook library.

The APR playbook library is structured in three tiers:
- Tier 1 (Universal): processes that exist in any enterprise (AP, IT incident management, customer support, recruiting, etc.)
- Tier 2 (Vertical-variant): processes that exist across industries but with industry-specific characteristics (e.g., procurement looks different in healthcare vs manufacturing)
- Tier 3 (Industry-native): processes specific to one industry (claims adjudication in insurance, KYC/AML in financial services, clinical trial management in pharma, etc.)

For V0, your library coverage is limited. You have access to Tier 1 patterns (universal) and partial Tier 2/3 patterns for three industries: financial services, healthcare/life sciences, manufacturing.

For each applicable process, output:
- Process name
- Tier classification
- Sub-process breakdown (where applicable)
- AI value pocket sizing (qualitative for V0: large/medium/small; quantitative for V0.5+)
- Reinvention archetype indication (Accelerate, Restructure, Retire, Invent)
- Prerequisites for AI investment (data, governance, change capacity)

Important for V0: if the prospect's industry is not in your Tier 2/3 coverage (financial services, healthcare/life sciences, manufacturing), return ONLY Tier 1 patterns and flag that Tier 2/3 coverage is forthcoming in V0.5.

This output is consumed by C2 Synthesis and C3 Quick Wins. Structure it for downstream consumption.

Return structured JSON only.
```

**User prompt template:**

```

- NAICS: {company_industry_naics}
- Size band: {company_size_band_estimate}

## Engineering implementation notes for Claude Code

### Prompt templating

Each agent's user prompt template uses standard placeholder syntax: `{variable_name}`. The orchestrator substitutes actual values. Recommended templating library: Jinja2 (Python) or equivalent.

### Output parsing

All agents return structured JSON. The orchestrator validates JSON structure against schemas (see Companion 05 Data Schemas) before passing to downstream agents. Validation failures trigger re-processing with feedback to the agent.

### Model API integration

Anthropic Claude API for V0. Architecture should support model swap (e.g., to Mistral for sovereign AI cases) by abstracting the model call layer. Recommended pattern: a Model interface with concrete implementations per provider.

### Token budget management

Each agent has a token budget (input + output). Long research outputs may exceed budgets; the orchestrator implements summarization fallback (Sonnet summarizes B-agent outputs if they exceed C2's context window).

### Latency tracking

Each agent's latency budget (specified in PRD) is tracked in metrics. SLA tracking surfaces budget overruns to orchestrator. 24-hour SLA decomposes into per-agent budgets that sum to <22 hours (leaving 2 hours for partner review and delivery).

### Cost tracking

Per-prospect cost is tracked at the agent level. Aggregated to per-prospect Diagnostic cost. Cost thresholds trigger kill switches (governance control from Hari's prior 100-instance failure lesson).

### Testing

The MeridianFS, NorthernCare, and AurelianTech demo scenarios from Companion 03 serve as integration test fixtures. Each agent should produce expected output (within tolerance) on the demo scenarios. Regression testing verifies agent behavior remains stable as prompts evolve.
